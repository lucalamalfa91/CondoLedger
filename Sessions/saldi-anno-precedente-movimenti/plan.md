# Piano — Saldi anno precedente (Movimenti, Situazione, PDF)

**Sessione**: `Sessions/saldi-anno-precedente-movimenti/`  
**Pattern**: Plan / Execute / Verify  
**Stato**: In attesa approvazione operatore

---

## Obiettivo

Introdurre la gestione autonoma dei **saldi dell'anno precedente** (apertura esercizio), separata da dovuti e versamenti, con:

1. Nuova sotto-sezione in **Movimenti**
2. Persistenza dedicata su Supabase
3. **Box dedicato** nella vista **Situazione**
4. **Box dedicato** nell'**export PDF**

---

## Decisioni di dominio (default proposti)

| Domanda | Decisione proposta |
|---------|-------------------|
| Cardinalità | **Un saldo per esercizio/immobile** (upsert) |
| Segno importo | **Positivo = debito riportato** (da pagare); **negativo = credito riportato** — allineato a colonna SALDO PREC. amministratore |
| Impatto totali preventivo/consuntivo | **Non** incrementa `preventivo`/`consuntivo` in `periodSummary`; resta voce separata |
| Totale da pagare | **Somma al totale da pagare** (`totale preventivo voci + saldo precedente`), ma **non modifica** l'importo iniziale preventivo/consuntivo |
| Presentazione UI/PDF | Box dedicato con copy esplicita: saldo positivo = **extra da pagare**; saldo negativo = **sconto / credito riportato** — sempre visibile come voce separata |
| Coesistenza riporti `carryFromPeriodId` | Flussi esistenti **restano**; warning UI se stesso esercizio ha sia `prior_balances` sia riporto su preventivo |
| Import documento | **Fuori scope** fase 1 (continua a creare dovuto riporto se configurato) |
| Esercizio provenienza | **Opzionale**, auto-suggerito con esercizio immediatamente precedente |
| Registro movimenti | Saldi precedenti **esclusi** dall'elenco dovuti/versamenti |
| Backup JSON | **Incluso** in schema v5 |

## User stories

### US-001 — Sezione Movimenti
Tab **Saldi precedenti** in Movimenti con lista + form (pattern Dovuti/Versamenti).

### US-002 — Inserimento/aggiornamento
Form: esercizio, importo (±), descrizione opzionale, provenienza opzionale. Upsert per esercizio.

### US-003 — Modifica/eliminazione
Edit/delete con conferma; persistenza Supabase; gestione auth.

### US-004 — Persistenza separata
Tabella `prior_balances` con RLS; load/save con reload pagina.

### US-005 — Box Situazione
Sezione **Saldi anno precedente** distinta da preventivo, consuntivo, versamenti, riporti. Chip riepilogo dedicato.

### US-006 — Box PDF
Stessa sezione in PDF preventivo e consuntivo quando valorizzato.

### US-007 — Separazione da dovuti/versamenti
Non compare in liste Dovuti/Versamenti né nelle tabelle rate/versamenti Situazione.

### US-008 — Totale da versare informativo
Box mostra saldo precedente + totale preventivo voci + totale da versare quando applicabile.

## Architettura

### Schema DB

```sql
create table prior_balances (
  id bigint generated by default as identity primary key,
  house_id bigint not null references houses(id) on delete cascade,
  fiscal_period_id bigint not null references fiscal_periods(id) on delete cascade,
  source_period_id bigint references fiscal_periods(id) on delete set null,
  amount numeric(12,2) not null,
  description text,
  created_at timestamptz not null default now(),
  unique (house_id, fiscal_period_id)
);
```

RLS: stesso pattern di `dues`/`payments`.
